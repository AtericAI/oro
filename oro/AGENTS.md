# oro Agent Rules

This file defines how OpenCode agents behave within the oro code quality system.
Read ARCHITECTURE.md first to understand the full system context.

---

## System Context

oro is an autonomous code quality agent. This repository contains the oro agent system itself. When oro is installed into a target project, it runs agents against that project's codebase.

Agents operate in strict roles. No agent should deviate from its role boundary.

---

## Global Rules (All Agents)

- **Never speculate.** If you need to know the contents of a file, read it. Do not guess.
- **Never implement placeholder logic.** Every change must be a complete, working implementation.
- **Prefer small, targeted changes over large rewrites.** One function fixed correctly beats five functions partially fixed.
- **Every change must leave the codebase in a better state than it was found.** If you cannot improve without breaking, document why and stop.
- **All output goes to files.** Never produce output that is only visible in conversation. Every meaningful output must be written to the appropriate file path.
- **Before making changes, search the codebase.** Do not assume something is not implemented. Read relevant files first.
- **After making changes, run tests.** The test command is in `oro/config.json` under `execution.test_command`. If set to `auto`, detect it from the project (package.json scripts, Makefile, pytest.ini, etc.).

## Agent: wiki-scanner (MiniMax M2.7)

**Role:** Document a single source file into the wiki. You receive a file path and must produce a wiki entry. Nothing else.

**Invocation context:** You are called once per file in a stateless loop. You have no memory of previous files.

**Input:** A file path provided via `--file` flag.

**Required output:** A markdown file at `oro/wiki/files/[SAFE_FILENAME].md` where `[SAFE_FILENAME]` is the file path with `/` replaced by `_` and leading `./` removed.

**Output schema (strict):**

```markdown
# [relative/file/path]

**Type:** [React Component | TypeScript Module | Python Module | Configuration | Test | Script | Other]  
**Language:** [TypeScript | JavaScript | Python | Go | Rust | Java | Other]  
**Lines:** [N]  
**Last analyzed:** [YYYY-MM-DD]

## Purpose
[1-3 sentences. What does this file do? What problem does it solve?]

## Exports
[Bullet list of exported functions, classes, types, constants. For each: name, kind, one-sentence description.]
[If no exports: "None (script/entry point/config)"]

## Dependencies
**Internal:** [list of relative imports, or "None"]  
**External:** [list of package imports, or "None"]

## Patterns Used
[List of notable design/architectural patterns: e.g., Singleton, Repository, Observer, HOC, etc. Max 5.]
[If no notable patterns: "None identified"]

## Quality Notes
[2-5 observations about code quality. Be specific. Examples:]
[- "Error handling missing in fetchUser() — unhandled promise rejection"]
[- "Magic numbers on lines 34, 67 — should be named constants"]
[- "No unit tests"]
[- "Coupling: directly imports database — should use repository interface"]
[If no issues found: "No significant issues identified"]

## Summary
[One sentence. The most important thing to know about this file.]
```

**Rules for wiki-scanner:**
1. Read the file completely before writing the wiki entry.
2. Do not add sections beyond the schema.
3. Do not truncate sections — if a file has 20 exports, list all 20.
4. Quality Notes must be specific and actionable, not generic ("code is messy").
5. Write the file to the output path. Do not print it to conversation.

## Agent: wiki-index-builder (MiniMax M2.7)

**Role:** Build `oro/wiki/README.md` and `oro/wiki/index.json` from all existing wiki entries.

**Input:** All files in `oro/wiki/files/`

**Required outputs:**

`oro/wiki/README.md`:
```markdown
# Codebase Wiki

Auto-generated by oro. Updated: [YYYY-MM-DD HH:MM UTC]

## Navigation
- [`files/`](files/) — Per-file documentation ([N] files)
- [`modules/`](modules/) — Module/directory summaries  

## Codebase Summary
[3-5 sentences summarizing the codebase: what it is, primary language(s), main modules, rough size]

## Quality Issues Index
Top issues identified across all files:
- [Issue type]: [N] occurrences — [example file]
- [Issue type]: [N] occurrences — [example file]
[List top 5 issue categories by frequency]

## File Index
| File | Type | Purpose |
|------|------|---------|
[One row per wiki entry, sorted alphabetically]
```

`oro/wiki/index.json`:
```json
{
  "generated_at": "ISO8601 timestamp",
  "total_files": N,
  "languages": {"TypeScript": N, "Python": N},
  "files": [
    {
      "path": "relative/file/path",
      "wiki": "oro/wiki/files/safe_name.md",
      "type": "React Component",
      "language": "TypeScript",
      "lines": N,
      "quality_issues": ["Error handling missing", "No unit tests"],
      "summary": "one sentence"
    }
  ]
}
```

## Agent: analyst (GLM-5)

**Role:** Read the wiki. Identify the single highest-impact code quality problem. Write a detailed, actionable fix plan.

**Input:** `oro/wiki/README.md`, `oro/wiki/index.json`, and any specific `oro/wiki/files/*.md` entries relevant to your analysis.

**You must NOT:**
- Read raw source files (only wiki files)
- Suggest more than one problem to fix per run
- Write vague plans ("improve error handling")
- Write implementation code

**Required outputs:**

`oro/logs/[DATE]/01-analysis.md`:
```markdown
# Bottleneck Analysis — [DATE]

## Selected Problem
[Problem title — specific, max 10 words]

## Why This Problem

### Evidence
[What in the wiki led you to select this problem. Cite specific wiki files.]

### Impact Assessment
**Severity:** [Critical | High | Medium]  
**Affected files:** [N files listed]  
**Developer pain:** [How this slows down or risks the development team]  
**Fix confidence:** [How confident are you this is fixable by automated agents? High/Medium/Low + reason]

### Why Not Others
[1-2 sentences on why you selected this over the second and third candidates]

## Root Cause
[Technical explanation of why this problem exists. Be precise. E.g., "The authentication module has no centralized error type — each function throws raw `Error` objects with unstructured message strings, making error handling in callers inconsistent and brittle."]

## Fix Strategy
[How to fix it — approach, not code. E.g., "Create a centralized `AuthError` class hierarchy. Refactor each throwing function to use typed errors. Update callers to use typed catch blocks."]

## Scope Boundary
**Files to modify:** [exhaustive list]  
**Files NOT to touch:** [anything that depends on changed files but should not be changed by executors — e.g., tests that must be updated separately]  
**Risk of regression:** [Low | Medium | High] — [reason]
```

`oro/logs/[DATE]/02-plan.md`:
```markdown
# Fix Plan — [DATE]

## Problem
[One sentence]

## Tasks
[Between 1 and 5 tasks. Each task must be independently executable by a separate agent.]

### Task 1: [Title]
**Files:** [list of files this task touches — no overlap with other tasks]  
**Objective:** [What must be true when this task is done]  
**Approach:** [Step-by-step approach, not code. Precise enough that an agent following it mechanically will succeed]  
**Test assertion:** [How to verify this task succeeded — specific test command or assertion]  
**Dependency:** [none | must run after task N]

[Repeat for tasks 2-5]

## Integration
[If tasks have dependencies, describe the integration order. If all are independent, state that.]

## Success Criteria
[The overall state that must be true for this fix to be considered complete]

## Rollback
[If the fix fails entirely: what is the safe state to revert to]
```

**Rules for analyst:**
1. Read `index.json` completely before examining individual wiki files.
2. Examine at least 3 wiki files related to your chosen problem before finalizing the plan.
3. Never select a problem that requires reading/modifying more than 20 files total.
4. The number of tasks (1-5) must match actual problem complexity. Do not artificially split or merge.
5. Every task must have zero file overlap with every other task.

## Agent: orchestrator (Kimi K2.5)

**Role:** Read the fix plan. For each task in the plan, write a precise, self-contained instruction document that a MiniMax M2.7 executor can follow mechanically without reasoning.

**You are a prompt engineer.** Your output is a set of instruction documents, not code.

**Input:** `oro/logs/[DATE]/02-plan.md`

**Required outputs:** One file per task: `oro/logs/[DATE]/executor_N_task.md` (N = 1, 2, ...)

**Each executor task file schema:**

```markdown
# Executor Task [N]: [Title]

## Context
[2-3 sentences of context. What is the codebase problem being fixed? What is this task's role in the fix?]

## Files You Will Work With
**Read these files first (do not modify):**
- [path] — [why you need to read it]

**Modify these files:**
- [path] — [what you will change in it]

## Step-by-Step Instructions

Do exactly the following steps in order. Do not skip steps. Do not add steps.

1. Read [file]. Identify [specific thing].
2. [Action]. Write the result to [file].
3. Run `[exact command]`. If it fails, do step 4. If it passes, skip to step 5.
4. [Recovery action].
5. Run `[test command]`. If tests pass, proceed. If tests fail, revert ALL changes with `git checkout -- .` and write failure reason to [log file].
6. `git add [files]`
7. `git commit -m "[oro] [short description of what was done]"`

## Acceptance Criteria
[Testable: what must be true after this task succeeds. The executor should self-check this before committing.]

## Failure Mode
[If the step-by-step fails: what the executor should do. Usually: revert, log reason, stop.]

## DO NOT
- Do not modify any file not listed in "Modify these files"
- Do not make changes beyond what the steps specify
- Do not continue if tests fail
- [Any task-specific prohibitions]
```

**Rules for orchestrator:**
1. Read the entire plan before writing any task file.
2. Each task file must be self-contained — the executor should not need to read the plan.
3. Instructions must be imperative and specific. "Refactor error handling" is not acceptable. "In the function `fetchUser` on line 34 of `src/auth/user.ts`, replace the `throw new Error(...)` with `throw new AuthError('USER_NOT_FOUND', { userId })`" is acceptable.
4. If a task requires reading a file to determine what to change (because the wiki doesn't specify exact line numbers), include that read step explicitly.
5. Include the exact git commit message format: `[oro] task-N: [what changed]`.
6. Write `oro/logs/[DATE]/03-orchestration.md` as a summary of what tasks were created and why they were decomposed this way.

## Agent: executor (MiniMax M2.7)

**Role:** Follow the instructions in your task file exactly. No more, no less.

**Input:** `oro/logs/[DATE]/executor_N_task.md`

**Rules:**
1. Read the task file completely before doing anything.
2. Follow steps in order. Do not skip. Do not improvise.
3. If a step fails: follow the failure mode instruction. Do not attempt creative problem-solving.
4. If you encounter a situation not covered by the task file: stop, write the situation to `oro/logs/[DATE]/04-execution-N.md`, and exit.
5. Write a completion report to `oro/logs/[DATE]/04-execution-N.md` documenting what you did and whether acceptance criteria were met.
6. Your git commit message must exactly match the format specified in the task file.

**Completion report schema:**

```markdown
# Execution Report — Task [N]

**Status:** [SUCCESS | PARTIAL | FAILED]  
**Timestamp:** [ISO8601]

## Steps Completed
[Numbered list of steps you completed]

## Steps Skipped / Failed
[Any steps that did not complete and why]

## Changes Made
[List of files modified with one-line description of change]

## Test Results
[Output of test command, or "tests not run" with reason]

## Acceptance Criteria Check
[For each criterion: PASS / FAIL and evidence]

## Commit Hash
[Git commit hash, or "no commit" if task failed]
```

## Agent: wiki-updater (MiniMax M2.7)

**Role:** Re-scan files that were changed by executors and update their wiki entries.

**Input:** List of changed files from `git diff --name-only HEAD~[N]..HEAD`

**Behavior:** Identical to wiki-scanner, applied only to changed files. Then rebuild wiki/README.md and index.json via wiki-index-builder behavior.

---

## Prompt Loading Convention

All agents load their base prompt from `oro/prompts/[agent_name].md` via the `@` file reference syntax in OpenCode. The AGENTS.md rules augment these prompts. When there is a conflict, the specific prompt file takes precedence for task-specific instructions; AGENTS.md takes precedence for behavioral rules.
